# HyMeKo: A Hypergraph Modeling and Computation Framework

## System Overview

---

### What HyMeKo Is

HyMeKo is a compiler-based framework for describing, analyzing, and computing over directed hypergraphs. It takes a human-readable text description of a system — where nodes represent entities and hyperedges represent multi-body relationships — and compiles it into deterministic, content-addressable intermediate representations and sparse tensor formats suitable for algebraic analysis, machine learning, and real-time inter-process communication.

The framework spans five Rust crates (~7,300 lines of library code), a comprehensive test suite (~8,000 lines), Python bindings via PyO3, and an IPC daemon for zero-copy tensor streaming. It is accompanied by 21 example data files, 25 architecture documents (Mermaid + SysML), and formal mathematical specifications.

### Capabilities

#### 1. Description Language

HyMeKo defines a concise, LALR-parseable language for labeled directed hypergraphs with the following constructs:

**Nodes** represent entities — physical links, sensors, neural network layers, states, or any domain object. Nodes support inheritance (`: base`), annotation tags (`<tag1, tag2>`), scalar/list/string values (`mass 25.0;`), and nested children (containment hierarchy).

**Edges** (prefixed with `@`) represent multi-body relationships — joints, connections, data flows, activations. Each edge contains hyperarcs with signed references:

- `+ref` — positive/source/outgoing (e.g., parent link in a joint)
- `-ref` — negative/target/incoming (e.g., child link in a joint)
- `~ref` — neutral/symmetric (e.g., shared frame)

References can carry weight annotations: `[[x,y,z],[r,p,y]] -child_link` encodes a spatial transform on the incidence connection itself.

**Imports** allow multi-file modular descriptions. A robot description imports a meta-kinematics schema; a neural network imports a layer metamodel. The compiler resolves imports recursively with cycle detection.

**Proven properties** (from the doctoral dissertation):

- The language is provably more concise than XML, JSON, and YAML for equivalent hypergraph descriptions (Thesis 1.2)
- Descriptions are in one-to-one correspondence with the associated hypergraph structures (Thesis 1.3)
- The tensor transformation can be executed in polynomial time (Thesis 2.4)

#### 2. SIMD-Accelerated Parser

The lexer provides three compile-time monomorphized implementations selected at runtime based on CPU feature detection:

- **AVX2** — processes 32 bytes per cycle for whitespace skipping and identifier scanning
- **SSE2** — processes 16 bytes per cycle as fallback
- **Scalar** — byte-at-a-time for non-x86 architectures

The parser is generated by LALRPOP (LALR(1)) and operates on zero-copy token references borrowed directly from the input buffer. No string allocation occurs during lexing of identifiers.

#### 3. Deterministic Intermediate Representation

The IR is a flat, arena-based structure with the following properties:

- **Unified declaration arena** — all nodes, edges, and arcs share a single `Vec<DeclNode>` with O(1) random access via `DeclId`
- **B-tree-ordered index** — fully qualified paths are stored in a `BTreeMap`, guaranteeing deterministic iteration order across compilations
- **Parent-child linked lists** — first_child / next_sibling pointers enable O(children) traversal without allocating child vectors
- **Jump tables** — `decl_to_node`, `decl_to_edge`, `decl_to_arc` vectors provide O(1) type-specific lookup from any DeclId
- **Signed references** — every arc reference carries its sign (+/−/~), target DeclId, optional annotation, and optional weight vector
- **Content-addressable hashing** — Blake3 Merkle hashes per declaration and a single canonical program hash enable O(1) structural change detection

The IR is serializable via `serde` and transportable as CBOR for persistent storage and inter-process exchange.

#### 4. Tensor Representations

From the compiled IR, the framework builds a CSR-based bipartite incidence view (`HyperGraphView`) and produces two sparse tensor formats:

**Star Expansion** — O(|E|·d) complexity. Preserves the hypergraph's bipartite incidence structure. Produces a 3D COO tensor of dimensions (|E| × (|V|+|E|) × (|V|+|E|)), where each slice k corresponds to one hyperedge. The sign semantics map to directionality: `+` places entries in the node→edge quadrant, `−` in the edge→node quadrant, `~` in both.

**Clique Expansion** — O(|E|·d²) complexity. Projects hyperedges into pairwise adjacency. Produces a 3D COO tensor of dimensions (|E| × |V| × |V|), where each slice k is the complete clique for hyperedge k. Sign pairs determine arc direction in the projected graph.

Both representations support:

- **Generic incidence values** — the `IncVal<F>` trait allows scalar weights, B-spline control points, fuzzy membership values, or any custom incidence type
- **Configurable aggregation** — `SignAgg` (PreferNonNeutral, WeightedVote, Channels3) and `WeightAgg` (Sum, Max, ProbabilisticSum, ŁukasiewiczSaturation) control how duplicate incidences are resolved
- **Parallel sort-reduce** — Rayon-accelerated sorting for large hypergraphs (threshold: 8,192 entries)
- **Degree-normalized variants** — for spectral methods requiring normalized Laplacians

Tensor formats available: COO (coordinate), CSR (compressed sparse row), nalgebra dense, nalgebra-sparse CSR, and JAX BCOO.

#### 5. Message Passing

The implicit clique step computes y = B·W·Bᵀ·x without materializing the dense matrix:

1. **Gather** — x_edges = Bᵀ·x_nodes (node signals aggregated into hyperedges)
2. **Scatter** — y_nodes = B·x_edges (hyperedge signals distributed back to nodes)
3. **Diagonal removal** — optional self-loop subtraction via precomputed diagonal

This is the computational kernel for Hypergraph Neural Networks (HGNNs) and the foundation for the HyperKAN architecture.

#### 6. Query Engine (designed, implementation ready)

The query engine evaluates arbitrary predicate trees against the compiled IR. Predicates compose recursively:

- **Element-level** — `Kind(Node|Edge)`, `Name(exact|prefix|contains|suffix)`, `InheritsFrom(base)`, `HasTag(tag)`, `HasValue(comparator)`, `MaxDepth(n)`
- **Structural** — `HasChild(predicate)`, `HasParent(predicate)`, `HasArcRef { sign, target }`, `ArcRefCount { matching, min, max }`, `ReferencedBy { sign, edge }`
- **Logical** — `And`, `Or`, `Not`, `Any`

The engine is domain-agnostic — it knows nothing about robotics, neural networks, or any application. Domain knowledge is expressed entirely as predicate patterns.

**Query-as-HyMeKo:** The same LALRPOP grammar that describes hypergraphs can describe patterns over hypergraphs. A query is a partial `.hymeko` fragment interpreted as a predicate tree. This unifies the description language and the query language under a single grammar.

#### 7. Domain Transformations (designed, implementation ready)

Query results feed into domain-specific transforms:

| Target | Input pattern | Output |
|--------|--------------|--------|
| **URDF** | Links + joints with signed incidence topology | XML robot description for ROS/Gazebo |
| **SDF** | Links + joints + simulation plugins | Gazebo SDF with physics configuration |
| **ROS2 launch** | Controllers + topic edges | Python launch files |
| **PyTorch nn.Module** | NN layer nodes + connection edges | Python class definition |
| **glTF** | Links + geometry + transforms | 3D scene graph |
| **HyperKAN input** | Kinematic links + joints | Star expansion COO tensor |

Each transform consists of a pattern set (a `.hymeko` query file) and an emitter. Adding a new domain requires zero changes to the engine.

#### 8. Structural Entropy (designed, implementation ready)

From the dissertation (Thesis 4), the framework computes:

**Normalized Laplacian** — L̂(H) = (D(H) − A(H)) / Σ D(x), where A(H) is the aggregated clique adjacency and D(H) is the degree matrix.

**Algebraic entropy** — I(H) = −Σ λ̂ᵢ log₂(λ̂ᵢ), where λ̂ᵢ are eigenvalues of L̂(H).

**KL divergence** — D_KL(H₁, H₂) between consecutive training states, measuring structural change between optimization steps.

These metrics feed into neural network training as a regularization term: J(θ_{t+1}) = J_t(θ) + α·D_KL(H_t, H_{t+1}).

#### 9. Zero-Copy IPC Streaming

The daemon crate uses iceoryx2 shared memory for inter-process tensor exchange:

- **Topic separation** — Star expansion stream, clique expansion stream, and query ingress on isolated topics
- **Memory loan injection** — Payloads are written directly into shared memory without buffer copying, achieving sub-100μs injection latency
- **Structural change detection** — The canonical program hash enables O(1) classification of topology changes (requiring full schema rebuild) vs. weight-only changes (requiring only pointer bumps)
- **Double-buffered state gate** — Atomic front/back buffer flipping prevents torn reads during concurrent producer-consumer execution

The Python subscriber reads tensors directly from shared memory via Apache Arrow's zero-copy C Data Interface, bypassing all serialization.

#### 10. Python Bindings

PyO3-based bindings expose the full pipeline:

- `PyHypergraphEngine` — factory for loading, parsing, and compiling `.hymeko` files
- `PyHypergraphIR` — compiled IR with name resolution, annotation access, and CBOR serialization
- `PyTensorCoo3D` — 3D sparse tensor with zero-copy Arrow export to PyTorch
- `PySparseMatrix2D` — 2D sparse matrix with zero-copy Arrow export
- `PySharedExpansion` — direct shared-memory tensor access from iceoryx2 segments

#### 11. HyperKAN Architecture (designed, implementation ready)

The Hyper-KA Isomorphism maps the Kolmogorov-Arnold representation theorem onto hypergraph convolution:

**Inner composition** (node → edge): h_e = Σ_{v∈e} φ_{e,v}(x_v), where φ is a learnable NURBS curve on each incidence connection.

**Outer composition** (edge → node): x'_v = Σ_{e∈E(v)} Φ_{v,e}(h_e), where Φ is a learnable NURBS curve on the return projection.

NURBS activations (Non-Uniform Rational B-Splines) replace standard B-splines, enabling exact representation of conic sections. The rational weights introduce a geometric degree of freedom that standard KANs lack.

The incidence topology is compiled once from a `.hymeko` file (e.g., a robot description) and reused for every forward pass. Only the NURBS control points and rational weights are learned during training.

### Application Domains

**Robotics** — Kinematic chains as hypergraphs where joints are multi-body hyperedges. A single `.hymeko` file describes links, joints, sensors, controllers, and communication topics. Query-based transformation produces URDF, SDF, and ROS2 configurations from the same source.

**Neural Network Architecture** — Network layers as nodes, connections as hyperedges. The clique and star expansions produce adjacency and incidence tensors for structural analysis. Entropy feedback regularizes training. NURBS-based HyperKAN learns on the compiled topology.

**Cognitive Systems** — Following the CogInfoCom framework, HyMeKo represents cognitive entities and their capabilities as hypergraphs with ontological hierarchy, enabling cross-domain modeling of perception, actuation, communication, and decision-making.

**Virtual Reality** — Scene graphs and multi-body simulations described as hypergraphs, with kinematic constraints as hyperedges connecting rigid bodies, frames, and interaction points.

**Knowledge Representation** — Many-to-many relationships expressed natively as hyperedges. The tensor representation enables algebraic operations (spectral clustering, centrality analysis, community detection) on knowledge graphs without projecting to pairwise graphs.

**Event Detection** — Directed hyperedges as event templates with precondition tails and consequence heads. Runtime activation and deactivation modeled as structural changes in the hypergraph, with observer-based dispatch.

### Design Principles

**Single source of truth** — One `.hymeko` file describes a system. All views (tensors, URDF, PyTorch code, visualizations) derive from it.

**Compilation over interpretation** — Parsing, resolution, lowering, and hashing happen once. Downstream consumers operate on the compiled IR.

**Zero-copy end-to-end** — From SIMD lexer tokens (borrowed from the input buffer) through Arrow-backed tensors (shared memory pointers) to PyTorch sparse tensors (DLPack handshake), no unnecessary copies occur.

**Generics over specialization** — The `IncVal<F>` / `EdgeWeight<V, F>` / `Real` trait system makes the tensor pipeline generic over incidence value types. Scalars, B-splines, NURBS, and future types plug in without changing existing code.

**Query-as-description** — The same grammar that describes hypergraphs describes patterns over hypergraphs. The query engine interprets partial `.hymeko` fragments as predicate trees.

**Content-addressable identity** — Blake3 Merkle hashes give every declaration and every compiled program a deterministic, collision-resistant identity. Structural changes are detected in O(1).
